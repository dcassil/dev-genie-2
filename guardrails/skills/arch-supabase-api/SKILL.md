# Supabase API — `supabase-api`

Source: `architectures/supabase-api/` (README.md, eslint.config.mjs, tsconfig.json). Database-as-API: Postgres is the source of truth (schema, constraints, RLS, RPC); Edge Functions are thin shims.

## What it provides

- **Spec** — schema → RLS → RPC → Edge Function layering, with append-only migrations and generated DB types.
- **`eslint.config.mjs`** (for Edge Function TS):
  - `no-restricted-imports` blocks `Deno.env.get` outside `_shared/env.ts`, sibling-function imports, and service-role client misuse.
  - `no-restricted-syntax` forbids `export *` and direct `Deno.env.get(...)` outside the env module.
  - `max-lines` 200 / 150-warn; `complexity` ≤ 10; `max-depth` ≤ 4; `max-params` ≤ 4.
  - `no-console` warn; `@typescript-eslint/no-floating-promises`, `no-explicit-any`.
- **`tsconfig.json`** — `strict`, `noUncheckedIndexedAccess`.

## Key rules

- Every table MUST `ENABLE ROW LEVEL SECURITY` in the same migration that creates it, with at least one policy.
- Service-role key lives ONLY in Edge Functions; client bundles use the anon key + RLS.
- Migrations are append-only — never edit a shipped migration.
- `functions/<a>/` MUST NOT import from `functions/<b>/`. Cross-function code goes in `_shared/`.
- All env via `_shared/env.ts`. `database.ts` is generated by `supabase gen types`, never hand-edited.

## Copy steps

```bash
SRC=<path-to-this-repo>/architectures/supabase-api
cp "$SRC/eslint.config.mjs" "$SRC/tsconfig.json" ./
```

Required directories: `supabase/migrations/`, `supabase/functions/_shared/`, `tests/rls/`, `types/database.ts`.

## Tooling

```bash
# DB
supabase db lint
supabase test db   # pgTAP — must cover allowed AND denied paths per table

# Edge Functions
deno check && deno lint && deno fmt
```

## Peer deps

```bash
npm i -D eslint eslint_d @eslint/js typescript-eslint globals
```

## When NOT to use

Heavy domain logic that doesn't fit in SQL/PLpgSQL — use `arch-node-api`. Long-running jobs (Edge Functions have execution limits).
